# 为什么选择这些功能？

> 通过企业 AI 挑战解释 Gas Town 的架构

## 问题所在

你有 AI 代理。也许有很多。它们在编写代码、审查 PR、修复 Bug、添加功能。但你无法回答基本问题：

- **谁做了什么？** 哪个代理写了这段有问题的代码？
- **谁可靠？** 哪些代理持续交付高质量成果？
- **谁能做这个？** 哪个代理应该处理这个 Go 重构？
- **什么相互关联？** 这个前端变更是否依赖某个后端 PR？
- **全局怎样？** 项目跨 12 个仓库的整体状况如何？

传统工具帮不上忙。CI/CD 跟踪构建，而非能力。Git 跟踪提交，而非代理性能。项目管理跟踪工单，而非谁真正做了什么以及做得怎样的微妙现实。

## 解决方案：工作账本

Gas Town 将工作视为结构化数据。每个操作都被记录。每个代理都有工作记录。每项工作都有来源。

这不是监控。这是**可见性** — 你从任何严肃的工程系统中都期望拥有的同种可见性。

---

## 功能：实体跟踪和归属

**问题所在**：你在 10 个项目中部署了 50 个代理。其中一个引入了关键 Bug。是哪个？传统的 git blame 显示通用的 "AI Assistant" 或更糟 — 人类的名字。

**解决方案**：每个 Gas Town 代理都有独特身份。每个操作都有归属：

```
Git 提交：    gastown/polecats/toast <owner@example.com>
Beads 记录：  created_by: gastown/crew/joe
事件日志：    actor: gastown/polecats/nux
```

**为什么重要：**
- **调试**：追踪问题到特定代理
- **合规**：SOX、GDPR、企业政策的审计追踪
- **问责**：确切知道谁在何时触碰到什么

---

## 功能：工作历史（代理 CV）

**问题所在**：你想分配一个复杂的 Go 重构。你有 20 个代理。一些擅长 Go。一些从未碰过。一些不稳定。你怎么选择？

**解决方案**：每个代理累积工作历史：

```bash
# 这个代理做了什么？
bd audit --actor=gastown/polecats/toast

# 在 Go 项目上的成功率
bd stats --actor=gastown/polecats/toast --tag=go
```

**为什么重要：**
- **绩效管理**：关于代理可靠性的客观数据
- **能力匹配**：将工作路由给经过验证的代理
- **持续改进**：识别表现不佳的代理以进行调优

这在 **A/B 测试模型** 时特别有价值。在类似任务上部署 Claude vs GPT，跟踪它们的完成率和质量，做出明智的决策。

---

## 功能：基于能力的路由

> **状态：计划中** — 技能跟踪和自动路由尚未实现。工作分配目前通过 `gt sling` 手动完成。

**问题所在**：你有 Go、Python、TypeScript、Rust 的工作。你有能力各异的代理。手动分配无法扩展。

**解决方案**：工作携带技能要求。代理有展示的能力（从其工作历史中得出）。匹配是自动的：

```bash
# 代理能力（从工作历史中得出）
bd skills gastown/polecats/toast
# → go: 47 tasks, python: 12 tasks, typescript: 3 tasks

# 基于匹配度路由
gt dispatch gt-xyz --prefer-skill=go
```

**为什么重要：**
- **效率**：合适的代理做合适的任务
- **质量**：代理在其擅长领域工作
- **规模**：分配上无人工瓶颈

---

## 功能：递归工作分解

**问题所在**：企业项目很复杂。一个"功能"变成跨 8 个仓库涉及 4 个团队的 50 个任务。平面问题列表无法捕获这种结构。

**解决方案**：工作自然分解：

```
Epic: User Authentication System
├── Feature: Login Flow
│   ├── Task: API endpoint
│   ├── Task: Frontend component
│   └── Task: Integration tests
├── Feature: Session Management
│   └── ...
└── Feature: Password Reset
    └── ...
```

每一层都有自己的链。汇总是自动的。你始终知道自己处于什么位置。

**为什么重要：**
- **可见性**：见林也见树
- **协调**：依赖关系明确
- **进度跟踪**：每一层的准确状态

---

## 功能：跨项目引用

**问题所在**：你的前端在后端 API 落地之前无法发布。它们在不同的仓库。传统工具不跟踪这种关系。

**解决方案**：显式的跨项目依赖：

```
depends_on:
  beads://github/acme/backend/be-456  # Backend API
  beads://github/acme/shared/sh-789   # Shared types
```

**为什么重要：**
- **无意外**：你知道什么在阻碍你
- **协调**：团队看到自己对其他团队的影响
- **规划**：基于实际依赖的现实进度

---

## 功能：联邦

> **状态：计划中** — 通过 Highway Operations Protocol (HOP) 的联邦已设计但尚未实现。Gas Town 目前作为单镇系统运行。

**问题所在**：企业项目跨多个仓库、多个团队，有时跨多个组织（承包商、合作伙伴）。可见性是碎片化的。

**解决方案**：互相引用的联邦式工作空间：

```bash
# 注册远程工作空间
gt remote add partner hop://partner.com/their-project

# 跨工作空间查询
bd list --remote=partner --tag=integration
```

**为什么重要：**
- **企业规模**：不受限于单仓库思维
- **承包商协调**：跟踪委托的工作
- **分布式团队**：尽管仓库分离仍拥有统一视图

---

## 功能：验证和质量门

**问题所在**：代理说"完成了"。真的完成了吗？代码质量可接受吗？通过审查了吗？

**解决方案**：带归属的结构化验证：

```json
{
  "validated_by": "gastown/refinery",
  "validation_type": "merge",
  "timestamp": "2025-01-15T10:30:00Z",
  "quality_signals": {
    "tests_passed": true,
    "review_approved": true,
    "lint_clean": true
  }
}
```

**为什么重要：**
- **质量控制**：不信任，要验证
- **审计追踪**：谁在何时批准了什么
- **流程强制执行**：门是数据，不仅是政策

---

## 功能：实时活动流

**问题所在**：复杂的多代理工作是不透明的。直到完成（或失败）之前你不知道正在发生什么。

**解决方案**：工作状态作为实时流：

```bash
bd activity --follow

[14:32:08] + patrol-x7k.arm-ace bonded (5 steps)
[14:32:09] → patrol-x7k.arm-ace.capture in_progress
[14:32:10] ✓ patrol-x7k.arm-ace.capture completed
[14:32:14] ✓ patrol-x7k.arm-ace.decide completed
[14:32:17] ✓ patrol-x7k.arm-ace COMPLETE
```

**为什么重要：**
- **实时调试**：问题发生时就看到
- **状态感知**：始终知道什么在运行
- **模式识别**：发现瓶颈和低效

## 企业价值主张

Gas Town 是开发者工具 — 像 IDE，但是用于 AI 编排。然而，其架构提供了企业级基础：

| 能力 | 开发者收益 | 企业收益 |
|------|-----------|---------|
| 归属 | 调试代理问题 | 合规审计 |
| 工作历史 | 调优代理分配 | 绩效管理 |
| 技能路由 | 更快完成任务 | 资源优化 |
| 联邦 | 多仓库项目 | 跨组织可见性 |
| 验证 | 质量保证 | 流程强制执行 |
| 活动流 | 实时调试 | 运营感知 |

**对于模型评估**：在可比任务上部署不同模型，客观跟踪结果，对在哪里使用哪种模型做出数据驱动的决策。

**对于长期项目**：不仅看代理在单个任务上的表现，还看其在复杂、多阶段、跨职能项目中的表现。

**对于跨职能团队**：跨仓库、团队甚至组织的统一可见性。

## 设计哲学

这些功能不是后加上去的。它们是基础性的：

1. **归属不是可选的。** 每个操作都有执行者。
2. **工作就是数据。** 不仅是工单 — 结构化的、可查询的数据。
3. **历史很重要。** 工作记录决定信任。
4. **规模是假设前提。** 从第一天起就是多仓库、多代理、多组织。
5. **验证优于信任。** 质量门是一等原语。

Gas Town 的构建是为了回答企业将在 AI 代理成为其工程工作流核心时提出的问题。
