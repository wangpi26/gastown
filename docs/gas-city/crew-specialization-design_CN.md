# Crew 专业化与基于能力的派发

**Bead：** hq-q76
**日期：** 2026-03-10
**参与者：** Mayor, Overseer
**相关：** beads `agent-cost-optimization` 分支, w-gc-001, w-gc-002, w-com-005, PR #2518, PR #2527

## 背景

三条工作线索汇聚到同一个设计问题：Agent 工作应当如何标记、匹配和路由到专业化的工作者？

1. **Beads `agent-cost-optimization` 分支** — 添加基于层级的派发（tool/basic/standard/advanced/human）和成本感知路由到 beads。层级系统是能力匹配的 80/20 代理。该分支的 §13 概述了泛化方向：基于能力的派发，其中任务声明需求，执行者声明能力档案。

2. **Gas City 角色格式（w-gc-001, w-gc-002）** — 计划中的声明式层，将 Gas Town 的硬编码角色形式化为可移植的、用户可定义的 Schema。PR #2518 原型化了 TOML 解析器。PR #2527 添加了每个 Crew 的 Agent 分配。

3. **Wasteland 信誉（stamps）** — 多维证明，将能力声明植根于证据。完成获得验证和盖章，建立与 Rig 句柄关联的信誉信号。

本文档记录了关于这些系统如何协同工作的设计洞察讨论。

## 1. 细胞模型：Agent 即创业公司

### 计划经济的问题

Beads 分支（§4）假设中央调度器计算最优的任务-工作者分配：读取每个任务的层级，找到满足它的最便宜 Agent。这是作业车间模型——全局最优但需要所有工作者和所有任务的中央知识。

### 替代方案：分布式派发

不使用中央规划者，每个 Agent 是一个**微型 Town**：

- **向上广播能力** — 它能处理什么
- **向下委派** — 给子 Agent 和工具
- **做出内部分配决策** — 基于本地知识

管理者 Agent 的能力是它下面一切的**传递闭包**。Mayor 不需要 `native_skills: [security-review]`，如果它管理一个拥有此能力的 Crew 成员。每一层将内部委派决策隐藏在能力抽象背后。

### 分形模式

这在每个尺度上递归：

| 尺度 | 单元 | 广播 | 委派给 |
|------|------|------|--------|
| 联邦 | Wasteland Rig | Stamps、信誉 | 它的 Gas Town |
| Town | Mayor | Rig 能力 | Crew、Polecats |
| Crew | 专家 | 领域专长 | 子 Agent、工具 |
| 工作者 | Polecat | 任务完成 | 工具 |
| 工具 | CLI 命令 | 确定性输出 | 无 |

### 委派作为认知的替代

来自 beads §13.1："认知是一种元能力——在运行时派生其他能力的能力，以溢价为代价。"

在细胞模型中，**委派替代认知**。一个 Sonnet 层级的 Crew 成员加上正确的子 Agent 可以实现 Opus 单独实现的效果，成本更低：

```
security-lead (Sonnet, 15k tokens 自身工作)
├── dep-scanner (Haiku, 5k tokens)
│   └── cargo-audit (工具, 0 tokens)
└── code-reviewer (Sonnet, 12k tokens)
    └── semgrep (工具, 0 tokens)

总计: 32k tokens 以混合费率
对比 Opus 全部自己做: 50k tokens 以 Opus 费率
```

专家已经**预计算了分解策略**，不仅仅是领域知识。

## 2. 能力档案：声明 + 证据

### 没有通用分类法

能力标签的通用通用语言是自败的。分类法僵化的速度快于问题演化的速度。"security.network.cors" 在任务是"嵌入在 WebSocket 升级请求头中带有 mTLS 客户端证书的 JWT 令牌"时毫无帮助。

取而代之，每个部门用**自己的语言**发布能力广告，包括明确的负面空间：

```yaml
crew: api-gateway-security
handles:
  - CORS 配置和调试
  - CSP 头策略
  - API 密钥验证和轮换
  - 速率限制和滥用防护
does_not_handle:
  - 密码学原语（→ crypto 团队）
  - 用户身份/密码管理（→ identity 团队）
  - 应用级 RBAC（→ 所属服务）
example_tasks:
  - "用户在跨源 API 调用时收到 403"
  - "需要为合作伙伴集成添加新的允许源"
anti_examples:
  - "需要轮换 TLS 证书"（→ infra）
  - "实现基于角色的访问控制"（→ app 团队）
```

`example_tasks` 和 `anti_examples` 比任何分类法都更有价值——它们用问题的语言而非解决方案的语言。调度器对示例进行模式匹配。

### 双向知识

任务在请求者和解决者达成一致之前不算完成，但它们知道不同的东西：

- **请求者**知道：症状是否解决，如何用问题空间语言描述问题
- **解决者**知道：根本原因是什么，运用了什么能力，如何用解决方案空间语言描述方案

能力档案需要**两者**。以配对路由示例捕获：

```yaml
routing_examples:
  - symptom: "跨源 API 调用出现 403 错误"
    resolution: "CORS allow-origin 配置"
    cost: 8000 tokens
    complexity: single-domain
```

症状是未来调度器会看到的。解决方案证明能力。

### 档案是活文档

能力档案以**声明**开始——由创建部门的人撰写。随时间推移，系统将这些声明植根于证据。这个生命周期塑造了档案格式：

**一个部门最初只有声明。** 它列出 `handles`、`does_not_handle`、`example_tasks` 和 `anti_examples`。这些存储成本低、闲置时免费，足以用于初始路由。但声明是未经验证的——一个声称处理 CORS 的部门可能从未解决过 CORS 问题。

**证据通过四个渠道积累：**

1. **成功完成。** 路由到该部门的任务已被解决并被请求者接受。症状-解决对成为植根于真实工作的路由示例。如果解决在该部门声称的能力之外（一次"拓展完成"），声明范围暂时扩展。

2. **弹回。** 一个任务到达这里，部门正确地拒绝了——"不是我的问题，试试 identity 团队。"这锐化了 `does_not_handle` 边界。反过来，如果部门*试图*解决但失败了，这是声称能力弱于声明的信号。

3. **重新打开的任务。** 请求者接受了解决方案，但修复没有持久——问题复现了。这是最强的负面信号：该部门在该领域的能力比其声明暗示的更浅。

4. **客观指标。** 成本（tokens 消耗）、持续时间和弹回历史由系统观察，而非自报。它们提供独立于请求者或解决者声明的校准。

**声明向证据漂移。** 随时间推移，档案中的 `example_tasks` 和 `anti_examples` 从编写假设转变为真实路由结果。`handles` 和 `does_not_handle` 通过弹回发现的实际边界得到细化。一个听起来合理但从未完成匹配任务的部门存在证据缺口——路由系统应该将其与在同一领域有 20 次完成的部门区别对待。

这个生命周期促成了路由决策的**分级信任模型**：

| 证据级别 | 信任级别 | 路由行为 |
|---------|---------|---------|
| 仅声明（0 次完成） | 推测性 | 仅在没有已验证替代时路由 |
| 1-3 次完成 | 试探性 | 路由低风险任务以积累信号 |
| 4-10 次完成 | 运作性 | 正常路由 |
| 10+ 次完成 | 已验证 | 优先为此任务类型路由 |

实际后果：将关键工作路由到已验证部门，使用低风险任务为试探性部门积累证据（探索/利用权衡）。这也解释了为什么扩展已有部门几乎总是比创建新部门更好——已有部门拥有新部门没有的路由历史。

## 3. 路由、弹回与学习

### 弹回即学习

在值班系统中，工单从一个组弹到另一个组寻找正确的解决者。每次弹回增加路由知识："Auth 团队看了这个，不是 auth 的问题，这是他们的发现。"这感觉像失败，实际上是系统在**发现自身的路由表**。

每次弹回是一个训练样本：
- 弹回的部门学到了边界（anti_example）
- 工单从每次尝试中积累了诊断上下文
- 失败的声明 + 成功的解决细化了路由信号

失败模式是这些知识被困在单个工单中而非反馈到能力档案中。系统应该从弹回中提取路由信号并自动更新档案。

### 跨领域任务的单所有权

跨领域任务应该有**单一归属部门**来分解和委派，而非共享所有权。原因：

- 产生更干净的路由信号："A 通过协调 B 和 C 处理跨领域安全问题"比"A+B+C 解决了这个问题"更有用
- **协调能力**（分解、委派、集成测试）是归属部门档案上的真实能力
- 升级路径保持清晰

```yaml
routing_examples:
  - symptom: "合作伙伴 WebSocket 连接上的间歇性认证失败"
    resolution: "跨领域: CORS + mTLS + JWT 令牌生命周期"
    delegated_to: [crypto-team, identity-team]
    own_contribution: "诊断、分解、集成测试"
    complexity: cross-domain
    cost: 45000 tokens (own: 15000, delegated: 30000)
```

## 4. 部门生命周期

§2 确立了档案如何通过完成、弹回、重新打开的任务和客观指标从声明演变为证据。本节讨论塑造部门格局的管理决策。

### 优先编辑而非创建

在 Agent 领域，启动一个新部门几乎是免费的——不像人类组织中人数和预算提供了天然的抗重复摩擦。这意味着泛滥风险是真实的：三个部门有重叠的 CORS 能力、没有明确的路由偏好、没有经济压力来合并。

在创建新专家之前，检查与已有 Crew 成员是否有 70%+ 的能力重叠。**扩展现有的。** 原因在 §2：有路由历史的已有部门比只有声明的新部门路由更好。闲置的部门维护成本为零，但其积累的证据重建代价高昂。

### 定期合并

Mayor（或专门的组设计功能）定期审查部门格局：

- 合并重叠显著的闲置部门
- 归档从未成功完成过任务的部门
- 为反复收到错误路由任务的部门重写档案（§2 的证据渠道标记这些）

### 何时修改声明

证据积累（§2）是自动的——系统观察完成和弹回。但**声明修改**是一个有意识的决策。部门应在以下情况下更新其 `handles` 和 `does_not_handle`：

1. **拓展完成积累** — 一次拓展是试探性的；同一领域三次就足以扩展声明
2. **弹回模式稳定** — 重复同方向弹回意味着边界是真实的，应记录
3. **争议能力浮现** — 如果重新打开的任务聚集在特定声明周围，要么投资该能力，要么明确放弃它
4. **明确重组** — Mayor 因战略原因重组部门（进入安全加固阶段，转移项目优先级）

### 复杂度加权

完成中的证据并非均质的。解决了一次棘手的跨领域事故的部门证明了**分解能力**。十个简单的 CORS 工单证明了**CORS 可靠性**。这是不同派发决策的不同信号：

- 简单的常规任务 → 路由到有可靠性证据的部门
- 模糊的跨领域任务 → 路由到有分解证据的部门

谁标记复杂度？请求者和解决者都不能单独决定：
- 解决者标记运用了哪些能力（解决方案空间）
- 请求者确认问题已解决（验收）
- 系统观察成本、持续时间、弹回历史（客观指标）
- 时间揭示修复是否持久（重新打开 = 有争议）

## 5. 对 Gas City 角色格式（w-gc-001）的影响

角色格式应支持细胞模型。角色不是平面的能力档案——它是一个可以包含子角色的**递归结构**：

```yaml
role: security-lead
goal: 处理此 Rig 的安全相关工作
layer: crew

# 能力广告（编写声明 — 参见 §2 了解这些如何演变）
handles:
  - CORS 配置和调试
  - 安全审计协调
  - CSP 头策略
does_not_handle:
  - 密码学原语（→ crypto）
  - 用户身份管理（→ identity）
example_tasks:
  - "用户在跨源 API 调用时收到 403"
  - "auth 模块的安全审计"
anti_examples:
  - "轮换 TLS 证书"（→ infra）

# 执行档案
cognition: standard          # Sonnet 层级配合子 Agent 足够
tools: [cargo-audit, semgrep, CVE-lookup]
context_docs: [OWASP-top-10.md, project-security-policy.md]

# 委派（细胞模型）
sub_agents:
  - role: dependency-auditor
    cognition: basic
    tools: [cargo-audit]
  - role: code-reviewer
    cognition: standard
    tools: [semgrep]

# 证据（由路由历史系统填充 — 参见 §2）
# 非人工编写；从完成、弹回和客观指标构建
track_record:
  routing_examples: []       # 真实任务的 symptom+resolution 对
  proven_boundaries: []      # 从弹回确认的 does_not_handle
  completions: 0
  trust_level: speculative   # → tentative → operational → proven
```

### 关键设计决策

1. **没有通用分类法** — 部门用示例和反例以自己的语言发布广告
2. **负面空间是必需的** — `does_not_handle` 防止弹回
3. **子 Agent 是一等的** — 角色格式是递归的
4. **声明向证据演变** — 档案是活文档（§2）
5. **来自 beads 分支的认知层级被保留** — 作为委派树中每个节点的下限
6. **委派成本可见** — 角色知道子 Agent 执行成本

### 与已有工作的关系

| 组件 | 来源 | 状态 |
|------|------|------|
| 层级系统 | Beads `agent-cost-optimization` | 在分支上已实现 |
| Agent 注册表 | Beads `agent-cost-optimization` | 在分支上已实现 |
| TOML 角色解析器 | Gastown PR #2518 | 开放 PR |
| 每个 Crew 的 Agent 分配 | Gastown PR #2527 | 开放 PR |
| Agent 框架调研 | Gastown PR #2581 (w-gc-004) | 已合并 |
| Wasteland stamps | HOP 联邦 | 已上线 |
| 角色格式设计 | **w-gc-001** | **本文档为其提供参考** |

## 6. 开放问题

1. **委派应该递归多深？** 一个管理子 Agent 的 Crew 成员，子 Agent 又管理子子 Agent，这很强大但复杂。2-3 层足够吗？

2. **谁写初始角色定义？** Mayor？Overseer？专门的组设计 Agent？角色作者需要跨领域的可见性来避免重复。

3. **动态履历如何反馈到路由？** 调度器是在路由时查询履历，还是预计算到路由索引中？

4. **Wasteland stamps 如何与本地履历关联？** 一个 Rig 的 Wasteland 信誉应反映其部门的聚合履历，但映射不简单。

5. **最小可行版本是什么？** 带有递归子 Agent 和动态履历的完整细胞模型是愿景。什么是用 beads 层级系统和已有 Gas Town 原语交付价值的 80/20 切片？
